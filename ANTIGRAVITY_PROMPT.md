## PROJECT CONTEXT

This is a **frontend-only prototype**. There is no backend, no database,
no API calls. All "live" data is generated by a JavaScript simulation engine
that runs in the browser on a timer (setInterval). The goal is a working,
demoable app — not a production system.

**Stack (non-negotiable):**
- React 18 + Vite
- JavaScript (no TypeScript)
- Tailwind CSS
- React Router v6
- Recharts (charts)
- Leaflet + react-leaflet (map)
- react-hot-toast (notifications)
- React Context (global state — no Redux, no Zustand)

## WHAT TO BUILD

### Phase 1 — Foundation (do this first)
1. Scaffold the Vite + React project
2. Install all dependencies from TECH_STACK.md
3. Set up Tailwind CSS
4. Create the folder structure exactly as defined in TECH_STACK.md:
   ```
   src/
   ├── components/
   ├── pages/
   ├── simulation/
   ├── context/
   └── assets/
   ```
5. Build the SimulationContext (React Context) with initial state:
   - 6 venue zones, each with: name, density (0–100), waitTime (minutes), staffCount
   - A queue list: Entry Gate, Food Court, Restrooms, Medical
   - An alerts array
   - A simulationRunning boolean
   - An activeScenario string

6. Build the simulation engine (`src/simulation/engine.js`):
   - Runs on setInterval every 4 seconds
   - Randomly fluctuates zone density and queue wait times realistically
   - Generates alert events based on density thresholds:
     - density > 80 → Critical alert for that zone
     - density > 60 → Warning alert
   - Exports: `startSimulation(dispatch)`, `stopSimulation()`, `applyScenario(name, dispatch)`

7. Build `src/simulation/scenarios.js` with these presets:
   - `normal` → moderate density, short queues
   - `surge` → 2–3 zones spike to critical
   - `emergency` → one zone goes to 100%, emergency alert fires
   - `halftime` → food/restroom queues spike sharply
   - `vip` → entry gate and specific zones surge briefly

8. Build `src/simulation/mockData.js` with:
   - 6 named zones (e.g., North Stand, South Stand, Main Concourse, etc.)
   - 8 staff members with names, zones, statuses
   - Initial queue data
   - Venue SVG zone polygon coordinates (simple rectangular zones for the map)

### Phase 3 — Core Components

Build these reusable components (in `src/components/`):

**`ZoneCard.jsx`**
- Props: zone name, density %, wait time, trend
- Colour-coded border and badge: green (<40%), amber (40–70%), red (>70%)
- Shows a small horizontal density bar

**`AlertStrip.jsx`**
- Sticky top banner that appears when any active alert exists
- Red background for Critical/Emergency, amber for Warning, blue for Info
- Dismissible with ✕ button
- Reads from SimulationContext alerts array

**`QueueItem.jsx`**
- Props: location name, wait minutes, trend arrow (↑ ↓ →)
- Full-width card with a coloured left border based on wait severity

**`KPICard.jsx`**
- Props: label, value, icon, colour
- Big number display for the Admin dashboard KPI row

**`SimLog.jsx`**
- Auto-scrolling event log
- Shows the last 50 simulation events
- Monospace font, timestamps, colour-coded by event type

**`HeatmapOverlay.jsx`**
- Leaflet layer that colours each zone polygon based on density
- Uses the green→amber→red gradient from DESIGN_GUIDELINES.md
- Clicking a zone opens a popup with zone name, density %, wait time

## CONSTRAINTS

- **No TypeScript.** Plain `.jsx` and `.js` files only.
- **No backend.** All data comes from the simulation engine.
- **No external state libraries.** React Context + useState only.
- **No CSS files.** Tailwind utility classes only.
- **No heavy libraries.** Only the 9 dependencies listed in TECH_STACK.md.
- The simulation engine must clean up its interval on unmount (`clearInterval`).
- The simulation log must cap at 50 entries to stay performant.
- All sensitive/admin routes must redirect unauthenticated users to `/`.
